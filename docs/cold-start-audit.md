## Area: README / Documentation

### [README] Port mismatch between README and `saw serve --help`

- **Severity**: P0 (Blocker)
- **What happens**: The README states `--addr` defaults to `:8080`. Running `saw serve --help` shows the actual default is `localhost:7432`. A user who follows the README and then tries `curl localhost:8080` after starting the server will get connection refused and have no idea why.
- **Expected**: README and `--help` should agree on the default address.
- **Repro**:
  ```
  # README line 80 says:
  Flags: `--addr` (default `:8080`), ...

  # But serve --help says:
  -addr string   Listen address (default "localhost:7432")
  ```

---

### [README] No prerequisites section — new user has no idea what's required

- **Severity**: P1 (High)
- **What happens**: The README jumps straight to `go install` with no mention of required Go version, git, or the `claude` CLI (needed for `--backend cli`). A user on Go 1.21 will get a cryptic error from `go install` because `go.mod` requires Go 1.25. A Claude Max user trying the `cli` backend won't know they need the `claude` CLI installed.
- **Expected**: A "Requirements" or "Prerequisites" section listing: Go >= 1.25, git, and optionally `claude` CLI or `ANTHROPIC_API_KEY`.
- **Repro**: `go install github.com/blackwell-systems/scout-and-wave-go/cmd/saw@latest` on Go < 1.25 fails silently or with a confusing toolchain error.

---

### [README] "IMPL doc" is used immediately without definition

- **Severity**: P1 (High)
- **What happens**: The very first usage example (`saw scout --feature "add OAuth support"`) says it "produces an IMPL doc." Every subsequent command also takes `--impl`. The term "IMPL doc" is never defined in the README. A new user doesn't know if this is a file format, a convention, or something they have to write by hand.
- **Expected**: A one-paragraph explanation of what an IMPL doc is (a Markdown planning document generated by `saw scout` that drives all subsequent commands), where it lives by default, and what it looks like. Linking to an example would help.
- **Repro**: Read README.md top-to-bottom as a new user. The phrase "IMPL doc" appears in Usage before any explanation.

---

### [README] Jargon used without definition: waves, scaffolds, I1–I6, invariants, worktrees

- **Severity**: P1 (High)
- **What happens**: The README uses "waves," "scaffold," "worktrees," "I1–I6," and "interface contracts" without explanation. The Protocol Compliance table lists I1–I6 with one-line descriptions, but a new user doesn't know these are formal protocol guarantees or why they matter. "Wave" in particular is used everywhere (commands, state machine, dashboard) with no definition of what a wave is in this context (a group of parallel agents working on non-overlapping files).
- **Expected**: A brief Concepts or Glossary section (or inline definitions in the Overview) explaining: wave = a group of parallel agents; scaffold = shared type stubs created before agents run; worktree = isolated git working directory per agent; I1–I6 = the six protocol invariants.
- **Repro**: Read the Overview section. "Each agent gets an isolated git worktree; the orchestrator merges, verifies, and sequences waves automatically." — the reader doesn't know what a wave is yet.

### [README] `--auto` flag missing from top-level help text

- **Severity**: P2 (Medium)
- **What happens**: The top-level `saw --help` output shows `--impl` and `--wave` in its Flags section, which implies they are global flags. They are not — they belong to specific subcommands. Meanwhile `--auto` (one of the most important flags for non-interactive use) does not appear in the global help at all.
- **Expected**: The global help Flags section should either be removed (since these aren't global flags) or replaced with a pointer like "Run `saw <command> --help` for per-command flags."
- **Repro**:
  ```
  saw --help
  # Shows --impl and --wave as top-level flags

  saw wave --help
  # Shows -impl, -wave, -auto as wave-specific flags
  # --impl and --wave in the top-level help are misleading
  ```

## Area: CLI Behavior

### [CLI] Subcommand `--help` exits with code 1

- **Severity**: P1 (High)
- **What happens**: Running `saw wave --help`, `saw scout --help`, `saw status --help`, or any other subcommand with `--help` exits with code 1 and prints "flag: help requested" as a trailing error line. Scripts that check `$?` after showing help will treat it as a failure. The top-level `saw --help` correctly exits 0.
- **Expected**: `--help` on any subcommand should exit 0. The "flag: help requested" line is stdlib `flag` package behavior that leaks through and should be suppressed.
- **Repro**:
  ```
  saw wave --help; echo $?
  # Output ends with: "wave: flag: help requested"
  # Exit code: 1

  saw --help; echo $?
  # Exit code: 0
  ```

---

### [CLI] Subcommand flag style is `-flag` (single dash), top-level help shows `--flag` (double dash)

- **Severity**: P2 (Medium)
- **What happens**: The global `saw --help` documents flags as `--impl <path>` and `--wave <n>` (double-dash GNU style). Subcommand help uses Go's stdlib `flag` package which displays `-impl` and `-wave` (single-dash). Both forms work at runtime, but the inconsistency between the two help screens is confusing. A user copying from the README (`--impl`) runs the subcommand and sees the help showing `-impl`, and wonders if they've used the right flag.
- **Expected**: Consistent flag display across all help text, ideally double-dash to match the README and CLI conventions.
- **Repro**:
  ```
  saw --help           # shows --impl, --wave (double-dash)
  saw wave --help      # shows -impl, -wave, -auto (single-dash)
  ```

---

### [CLI] `saw --version` reports "saw dev" for locally built binary

- **Severity**: P2 (Medium)
- **What happens**: Running `saw --version` on a locally built binary prints `saw dev`. The README says "Print version" but doesn't mention that version injection requires ldflags. A new contributor who runs `go build ./cmd/saw/` (the documented build path) gets a useless version string.
- **Expected**: Either document that `make build` is required to inject version info, or make the fallback more informative (e.g., `saw dev (build with 'make build' for version injection)`). The Makefile does not pass `-ldflags` either — only GoReleaser does — so `make build` also produces "saw dev".
- **Repro**:
  ```
  go build ./cmd/saw/
  ./saw --version
  # Output: saw dev
  ```

---

### [CLI] Missing required flag errors give no usage hint

- **Severity**: P2 (Medium)
- **What happens**: Running `saw wave` with no arguments prints `wave: --impl is required` and exits 1. Running `saw scout` with no arguments prints `scout: --feature is required` and exits 1. Neither error shows the available flags or a pointer to `--help`. A new user sees the error and must guess that `saw wave --help` exists.
- **Expected**: After printing the missing-flag error, also print the subcommand usage or append "Run 'saw wave --help' for usage."
- **Repro**:
  ```
  saw wave
  # wave: --impl is required
  # (no usage shown)
  ```

## Area: Build / Setup

### [BUILD] No web UI build instructions for contributors — `make build` prerequisite is implicit

- **Severity**: P2 (Medium)
- **What happens**: The web UI (`web/dist/`) is embedded into the Go binary via `go:embed`. If `web/dist/` is absent or stale, `go build ./cmd/saw/` fails with an embed error. The README only documents `go install` (which pulls a pre-built release) and does not mention that contributors need to run `cd web && npm install && npm run build` (or `make build`) before `go build`. The `examples/` directory exists but is completely empty.
- **Expected**: A "Contributing / Building from source" section in the README explaining: (1) run `make build` to build the full binary including web UI, (2) requires Node.js for the web build step, (3) `go build ./cmd/saw/` alone will fail if `web/dist/` is missing.
- **Repro**:
  ```
  git clone <repo>
  go build ./cmd/saw/
  # Fails if web/dist/ not present — no explanation in README
  ```

### [BUILD] `go.mod` requires Go 1.25, but Go 1.25 does not exist

- **Severity**: P3 (Low)
- **What happens**: `go.mod` declares `go 1.25.0`. As of this audit the latest Go release is 1.26.0, and Go 1.25 was a real release — but the README lists no minimum Go version, so a user on Go 1.24 will hit a toolchain version error with no explanation.
- **Expected**: README should document the minimum Go version. Given `go.mod` says 1.25, the prerequisites section should say "Go 1.25 or later."
- **Repro**: `go.mod` line 3: `go 1.25.0`; README has no Go version requirement.

## Area: Discovery / Getting Started

### [DISCOVERY] No QUICKSTART, getting started guide, or worked example

- **Severity**: P1 (High) — overlaps with README jargon issue above but distinct
- **What happens**: The README goes Overview → Installation → Usage (six bare command examples) → Architecture → Protocol Compliance. There is no end-to-end walkthrough. A new user does not know the intended workflow: (1) run `saw scout` to generate an IMPL doc, (2) review it (optionally via `saw serve`), (3) run `saw wave --auto` to execute. The six code examples in Usage are not annotated to show they are sequential steps.
- **Expected**: A "Quickstart" or "Workflow" section that walks through one complete feature from `saw scout` to `saw wave --auto`, showing what files are created and what each step does. The `examples/` directory is empty and could house a sample IMPL doc.
- **Repro**: Read README.md as a new user and try to answer: "What do I actually do first?"

### [DISCOVERY] No explanation of relationship to the protocol repo

- **Severity**: P3 (Low)
- **What happens**: The README links to `https://github.com/blackwell-systems/scout-and-wave` in the first line and again in the Protocol Compliance section, but never explains what that repo is or why a user of this Go CLI should care about it. A new user may wonder if they need to install something from that repo, whether the protocol repo has usage docs that apply here, or if this Go repo supersedes it.
- **Expected**: A sentence clarifying the relationship, e.g.: "The protocol repo defines the SAW specification. This repo is the Go CLI implementation — you do not need to clone or install the protocol repo separately."
- **Repro**: Follow the link in line 10 of README.md — the protocol repo has no mention of this Go CLI.
